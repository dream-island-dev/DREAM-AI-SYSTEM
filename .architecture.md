## 1. High-Level Overview

A Hebrew-RTL, React 19 SPA deployed on Vercel, backed by Supabase (Postgres + Deno Edge Functions + Auth).  
Department managers log in via Google SSO, interact with a **tier-scoped AI agent** (Gemini-first, Claude fallback),  
generate shift schedules, upload operational data, and receive real-time push notifications.

### Multi-Agent System (MAS) Vision

The system operates as a **Hierarchical Multi-Agent System** where each user's agent is dynamically scoped  
to their `role_tier` — controlling which tools, data, and capabilities are exposed in the AI context.

## 3. Database Schema

### Core Auth Extension

**`profiles`** — extends `auth.users` 1:1
```
id (UUID, FK auth.users)  name  email  role  role_tier  department  avatar_text  avatar  status
```
- `role` CHECK: `('super_admin','admin','manager','staff')` — legacy auth gate
- `role_tier` CHECK: `('super_admin','general_manager','tactical_manager')` DEFAULT `'tactical_manager'` — **MAS tier** ← NEW (Migration 012)
- `department` CHECK: `('housekeeping','maintenance','reception','spa','management')` OR NULL
- Owner model: `tzalamnadlan@gmail.com` = super_admin · `promote7il@gmail.com` = general_manager (admin)
- `set_my_department(p_department TEXT)` — SECURITY DEFINER RPC for frontend updates
- `get_my_role_tier()` — SECURITY DEFINER helper used in RLS policies ← NEW (Migration 012)
- Backfill: `role='super_admin'→role_tier='super_admin'`, `role='admin'→role_tier='general_manager'`, all others→`'tactical_manager'`

### Agent Intelligence Layer

**`agent_profiles`** — one active profile per manager
```
id  manager_id (FK profiles)  department  display_name  system_prompt (TEXT)
personality_traits (JSONB)  drive_folder_url  questionnaire_id  is_active
```
- UNIQUE(manager_id) — one agent per manager, upserted on re-creation
- `system_prompt` is the AI-generated core of each agent

**`questionnaire_responses`** — raw questionnaire answers
```
id  manager_id  department  responses (JSONB)  drive_folder_url  agent_profile_id
```

**`chat_history`** *(live table name)* — per-session conversation log
```
id  session_id  agent_id  manager_id  role ('user'|'assistant')  content  created_at
```
- Note: `schema.sql` uses `conversation_history`; live DB uses `chat_history` (migration 001)

**`agent_learning_logs`** — manager feedback / corrections (few-shot injection)
```
id  agent_profile_id  manager_id  conversation_message_id
original_response  correction  rating  feedback_type  is_injected
```

**`agent_memory`** ← **NEW — Migration 011 (Pillar 3)**
```
id  manager_id (FK profiles)  department  rule_text  source_file_name
category  is_active  created_at
```
- Soft-delete via `is_active = false` (preserves audit trail)
- Populated by `process-knowledge` Edge Function via Gemini multimodal extraction
- Injected into: `chat` system prompt · `generate-agent-profile` · `generate-schedule`

### Operational Tables

| Table | Purpose | Key Fields |
|-------|---------|-----------|
| `employees` | Staff roster | name, department, role, phone, status |
| `shifts` | Shift schedule | employeeName, department, date, start, end, status |
| `guests` | Guest check-ins + **EZGO pipeline** | name, phone, room, room_type, arrival_date, status, manager_id, msg_pre_arrival_sent, msg_room_ready_sent, msg_post_checkin_sent |
| `notification_log` | WhatsApp dedup | guest_id, trigger_type, sent_at |
| `service_calls` | Maintenance tickets | title, priority, department, assigned_to, status |
| `checklist_tasks` | Daily checklists | task, department, done, task_date |

**`guests` — augmented fields (Migration 012):**
```
manager_id (UUID, FK profiles, nullable)  — auto-stamped on INSERT via BEFORE trigger
msg_pre_arrival_sent  (BOOLEAN, DEFAULT FALSE)  — EZGO step 1
msg_room_ready_sent   (BOOLEAN, DEFAULT FALSE)  — EZGO step 2
msg_post_checkin_sent (BOOLEAN, DEFAULT FALSE)  — EZGO step 3
```
**RLS on `guests` (replaces open `qual:true` policy):**
- `guests_super_admin_all` — super_admin: ALL on all rows
- `guests_gm_select` — general_manager: SELECT on all rows
- `guests_own_all` — any auth user: ALL on their own rows (manager_id = auth.uid())
- Auto-stamp trigger: `guests_set_manager_id` — BEFORE INSERT, sets manager_id = auth.uid() if null
- Index: `idx_guests_manager ON guests(manager_id)`

### Infrastructure Tables

**`push_subscriptions`** — Web Push subscription objects
```
id  user_id (FK profiles, UNIQUE)  subscription (JSONB)  user_agent  updated_at
```
- RLS: users manage only their own row
- Stale subs (410 Gone) auto-deleted by `push-notify` Edge Function

## 4b. MAS Logic Preparation — Edge Function Tier Injection (Phase 2)

### How `role_tier` reshapes the AI agent per user

Both `chat` and `generate-agent-profile` will fetch `role_tier` from `profiles` after auth,  
then inject a tier-specific **capabilities block** into the system prompt. This makes each  
manager's agent a genuinely different instrument — not just the same prompt with different data.

#### `chat/index.ts` — Proposed step 2c (after memory injection)

```typescript
// ── 2c. Fetch role_tier and inject tier capabilities ─────────────────────
const { data: tierProfile } = await supabase
  .from("profiles")
  .select("role_tier, department, name")
  .eq("id", managerId)
  .single();

const roleTier = tierProfile?.role_tier ?? "tactical_manager";

const TIER_CAPABILITIES: Record<string, string> = {
  super_admin: `
## הרשאות מערכת — רמת SUPER ADMIN
יש לך גישה מלאה לכל מחלקות המלון, כל הנתונים, וכל הגדרות המערכת.
כלים זמינים: ניתוח נתונים גלובלי, ניהול פרופילי סוכנים, בקרת הגרות, דוחות ROI כלל-מלוניים.
אל תגביל תשובות לפי מחלקה — ענה על שאלות רוחביות.`,

  general_manager: `
## הרשאות מנהל כללי — רמת GENERAL MANAGER
יש לך גישת קריאה רוחבית לכל מחלקות המלון.
כלים זמינים: ניתוח KPIs בין-מחלקתי, חישובי ROI, השוואת ביצועים, זיהוי מגמות.
אינך יכול לכתוב לנתוני מנהלים אחרים — אך תוכל לראות הכל.`,

  tactical_manager: `
## הרשאות מנהל טקטי — רמת TACTICAL MANAGER
אתה פועל בסביבה מבודדת למחלקת ${tierProfile?.department ?? "המחלקה שלך"}.
כלים זמינים: ניהול משמרות, pipeline אורחים (EZGO), שליחת WhatsApp, צ'קליסטים יומיים.
אל תגיש מידע על מחלקות אחרות — עבוד רק בתחום הרשאותיך.`,
};

systemPrompt += "\n\n---\n" + (TIER_CAPABILITIES[roleTier] ?? TIER_CAPABILITIES.tactical_manager);
```

#### `generate-agent-profile/index.ts` — Proposed step 2c (after memory section)

```typescript
// ── 2c. Resolve tier and inject into generation prompt ───────────────────
// role_tier is already in profiles — fetched via the same supabase client.
const { data: tierData } = await supabase
  .from("profiles")
  .select("role_tier")
  .eq("id", user.id)
  .single();

const roleTier = tierData?.role_tier ?? "tactical_manager";

const TIER_TOOLS: Record<string, string> = {
  super_admin:       "כלים: ניתוח נתונים גלובלי, ניהול מערכת, דוחות ROI, שליטה בתצורה.",
  general_manager:   "כלים: KPI רוחבי, מדדי ביצועים בין-מחלקתיים, ניתוח מגמות.",
  tactical_manager:  "כלים: ניהול משמרות, EZGO pipeline, WhatsApp, צ'קליסטים יומיים.",
};

// Injected into generationPrompt template:
// רמת הרשאות: ${roleTier} | ${TIER_TOOLS[roleTier]}
// This ensures the generated system_prompt bakes in tier awareness permanently.
```

### Key design rules
1. **Tier is fetched server-side** — never trusted from client request body (same pattern as `department` in `process-knowledge`)
2. **Tier block is appended LAST** in systemPrompt — after base prompt, Drive RAG, memory rules, learning-logs — so it acts as a hard override frame
3. **Backcompat** — if role_tier is null/missing, default to `'tactical_manager'` (most restrictive)
4. **whatsapp-cron** uses service_role → bypasses RLS entirely, no tier check needed

## 7. Pillar 3 — Agent Long-Term Memory (Current Implementation)

### Ingestion Pipeline (stateless — no Supabase Storage used)

```
Manager uploads PDF/PNG/JPG/TXT/DOCX
      │
      ▼ [Browser]
KnowledgeUploader.js
  ├── TXT      → FileReader.text()
  ├── DOCX     → FileReader.arrayBuffer() → mammoth.js (lazy) → plain text
  ├── PDF/IMG  → FileReader.arrayBuffer() → Uint8Array → btoa() → base64
  └── Size guard: > 15 MB → error toast
      │
      ▼ POST to process-knowledge Edge Function
        body: { fileName, mimeType, content, isText }
        header: Authorization: Bearer <user JWT>
      │
      ▼ [process-knowledge/index.ts]
  ├── Auth: supabase.auth.getUser(token) → manager_id
  ├── Fetch department from profiles
  ├── Gemini 1.5 Flash call:
  │     isText=true  → text part only
  │     isText=false → inline_data (mimeType + base64) + text prompt
  ├── Parse JSON: { rules: [{rule_text, category}] }
  ├── Batch INSERT into agent_memory
  └── Return { ok, rules_extracted, rules }
      │
      ▼ [KnowledgeUploader.js] displays rules + allows soft-delete
```

### Context Injection (Steps 5-7 — LIVE as of 2026-06-07)

```
agent_memory rows injected into:
  1. chat v12            — step 2b, limit 20 rules, by manager_id
                           Section: "## כללי עבודה שהמנהל לימד את הסוכן"
  2. generate-agent-profile v10 — step 2b, limit 15, by manager_id
                           Injected as bullet list before generation prompt separator
  3. generate-schedule v4 — step 2b, limit 10, by manager_id
                           Section: "כללים נוספים שהמנהל לימד את הסוכן"
                           ShiftGenerator.js passes managerId: user?.id in invoke body
```

### Memory Scoping Rules
- **Personal context** (chat, profile generation): query by `manager_id` — each manager's own uploads
- **Scheduling context**: query by `manager_id` + `department` — only this manager's scheduling rules

## 8. Data Flow: Department Locking

All shift and employee data is department-tagged at the application layer (not DB trigger):

```
AgentQuestionnaire → set_my_department RPC → profiles.department
        ↓
DataUpload.js    → on mount: SELECT department FROM profiles WHERE id = user.id
                 → overrides department field in all imported shift rows
ShiftGenerator.js → same fetch → passes department to generate-schedule
generate-schedule  → DEPT_RULES static map + (future) agent_memory dynamic rules
                   → stamps department on every output row server-side
```

## 9. Security Model

| Concern | Implementation |
|---------|---------------|
| `ANTHROPIC_API_KEY` | Supabase secret ONLY — never in `REACT_APP_*` |
| `VAPID_PRIVATE_KEY` | Supabase secret ONLY — never in frontend |
| `WHATSAPP_SIMULATION` | Hardcoded `true` until Meta approval |
| VAPID public key | Safe to expose in `REACT_APP_VAPID_PUBLIC_KEY` (by design) |
| DB access | RLS on all tables; Edge Functions use `service_role` internally |
| User auth | JWT verified in every Edge Function via `supabase.auth.getUser(token)` |
| Push subscriptions | RLS: `auth.uid() = user_id`; stale subs auto-cleaned on 410 |
